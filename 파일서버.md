## 1. 파일 저장 로직

1. 사용자에게서 파일을 업로드 받음( Admin모듈에서)
2. 받은 파일을 파일 서버(개인 오라클 서버)로 업로드 하면서 파일의 크기, 이름, 접근 경로 등을 받아옴
    - 파일 서버에서는 해당 파일을 로컬에 저장하고 파일의 정보를 반환 해줌
3. 받아온 정보를 우리 프로젝트 DB에 집어 넣음
4. 해당 정보를 통해 파일에 접근함

---

## 2. 문제점

1. 파일에 접근하는 주소의 패턴은 해당 파일을 저장하는 파일 서버의 DB의 PK값(시퀀스)로 접근을 해서, 파일 접근 경로의 규칙성이 존재
2. 저장한 파일에는 **보호자원** 이라는 개념이 없음 -> 온라인 강의 영상, 특정 강의의 교안, 관리자만 접근 가능해야 하는 기타 파일들 을 보호할 수 없음

---

## 3. 해야되는것

1. 파일에 보호자원이라는 개념을 얹어야 함

---

## 4. 해결방안

### 1. 방법

1. 파일을 저장할 때 해당 파일에 접근 가능한 사람들의 권한 또는 정보를 파일 서버에 같이 저장하고, 그 파일에 접근하는 요청이 왔을때 해당 사용자의 정보를 조회해서 접근 허가/거부 하기
2. 파일 서버의 파일 저장 로직은 그대로 냅두고, 파일 접근을 Admin 모듈을 통해서만 가능하게 하기 -> 근데 이건 1번보다 별로일 것 같음 그리고 리액트 쪽에서 접근하는것이 복잡해지고, 관리자만 접근 가능하게 만들어 놓은 서버를 굳이 유저가 접근하게 할 이유 또한 없음

#### a. 파일 저장 로직에 보호자원이라는 개념 얹기

1. 현재 파일을 업로드 하는건 Admin 모듈에서만 가능함.
2. 하지만 파일에 접근하는것은 일반 사용자(React + Rest모듈)에서도 발생함
3. 제일 베스트는 JWT토큰으로 인증/인가를 처리해서 파일에 접근할지 말지를 결정하는것
4. 하지만 지금 JWT토큰은 Rest모듈에서만 사용하고 있고(로직은 common에 있긴함), JWT토큰에는 지금 ROLE_USER, ROLE_ADMIN 과 사용자의 ID, 이름 정도만 담고 있음 -> 담는 정보를 추가해야 하고, Admin쪽에서도 JWT토큰을 발행해서 DB에 저장하는 등의 로직이 필요함
5. 또한, 전달받은 JWT 토큰의 유효성 검사와, 토큰에 담긴 사용자가 실재하는 사용자인지, 올바른 권한인지를 처리하는 로직이 파일서버에서 필요할 것이고, 이에 따라 현재 사용하는 최종 프로젝트 DB에도 파일 서버가 접근해야됨

#### b. 파일 서버는 단순 저장소로만 유지하고 권한 체크를 밖에서 하기

1. 위의 방법은 너무 복잡해지고 파일서버가 프로젝트의 DB까지 알아야 해서 결합이 심해짐
2. 파일에 접근할 때 특정한 API 키를 요구하게 만들고, 권한 체크를 Rest 모듈이나 어드민 모듈에서 진행한 뒤 유효하면 각 모듈에서 직접 파일 서버에 요청을 보낸 뒤 받은 데이터를 스트림 그대로 클라이언트에 전달한다.
3. 문제점:
    1. 각 모듈에 네트워크 부하가 생김
    2. 일반 파일은 크게 영향 없는 수준이겠지만, 동영상 강의 같은 경우에는 문제가 생길 수 있음. 100명이 동시에 강의를 시청한다면, Rest서버가 100개의 스트림을 동시에 파일 서버에서 받아서 클라이언트로 전송해야됨 -> 병목 발생
4. 동영상 파일 대안:
   **Pre-signed URL 패턴**: 1. 클라이언트 -> REST /api/files/{id}/signed-url (JWT 포함) 2. REST 모듈 -> 권한 체크 (구매 여부 등) 3. 통과하면 -> 단기 토큰 생성 (Redis 저장, 60초 만료) 4. 파일서버 URL + 토큰 반환: http://fileserver/files/{id}/view?token=xxxx ... 5. 클라이언트 -> 파일서버 직접 접근 (토큰 검증만)
   파일 서버는 토큰 만료 여부만 확인하면 되고, 동영상 스트림은 파일서버 -> 클라이언트 직접 전송

## 4-2. 파일 접근 흐름 정리

### 비보호 자원 (공개 이미지)

```
업로드: Admin → Cloudinary → URL 반환 → DB 저장
접근:   <img src="https://res.cloudinary.com/..."> 직접 사용 (인증 불필요)
```

### 보호 자원 — Admin Thymeleaf 접근

```
브라우저 → GET /admin/files/{fileId}/view
  ↓ AdminFileController: canAccess() 인가 체크
  ↓ AdminFileController: API 키로 파일서버에 직접 연결
  ↓ 파일서버 응답 스트림을 그대로 클라이언트로 전달 (프록시)
브라우저 ← 파일 스트림
```

### 보호 자원 — React + REST 접근 (Pre-signed URL)

```
클라이언트 → POST /api/files/{fileId}/token  (JWT 포함)
  ↓ FileController: 인증/인가 체크 (구매 여부, 수강 여부 등, 미구현)
  ↓ FileAccessTokenService: UUID 생성 → Redis에 직접 저장 (60초 TTL)
  ↓                          "https://paste.maerchen.dev/.../view?token={uuid}" 구성
클라이언트 ← Pre-signed URL 반환

클라이언트 → GET https://paste.maerchen.dev/api/storage/files/{fileId}/view?token={uuid}
  ↓ 파일서버: Redis에서 토큰 조회 → 있으면 통과, 없으면 401
클라이언트 ← 파일 스트림 (파일서버 → 클라이언트 직접, REST 서버 미경유)
```

> **핵심**: REST 모듈이 Redis에 직접 쓰고, 파일서버가 Redis에서 읽어 검증. 파일서버는 토큰 만료 여부만 확인하고 권한 로직은 모름.

## 4-3. 동영상 강의 스트리밍 문제

현재 TTL이 60초이기 때문에, 60분짜리 강의를 시청하는 도중 토큰이 만료됨.
`<video>` 태그는 seek(탐색) 시마다 Range 요청을 새로 보내는데, 그 시점에 토큰이 만료되어 있으면 재생이 끊김.

### 방안 비교

#### ① 동영상 전용 긴 TTL

강의 업로드 시 영상 길이(초)를 DB에 저장해두고, 토큰 발급 시 `max(강의길이 × 1.5, 3600)` 으로 TTL을 계산.

```
클라이언트 → POST /api/files/{fileId}/token
  ↓ FileController: 파일 타입 확인 (video/*)
  ↓ 영상 길이 조회 → TTL 계산 (예: 60분 강의 → 5400초 TTL)
  ↓ Redis에 토큰 저장 (긴 TTL)
클라이언트 ← Pre-signed URL 반환 → <video src> 세팅
```

- 장점: 구현 변경 거의 없음, 단순
- 단점: URL 유출 시 TTL 동안 누구든 접근 가능. 강의 길이를 업로드 시점에 알아야 함.
- **현실적인 최종 프로젝트 선택지**

---

#### ② 프론트 토큰 주기적 갱신

React에서 `setInterval`로 45초마다 토큰을 재발급받아 `<video src>`를 교체.

```
최초: POST /api/files/{id}/token → src 세팅
45초마다: POST /api/files/{id}/token → src 교체 시도
  → <video>.src 교체 시 재생 위치 초기화 문제 발생
  → currentTime을 저장했다가 복원하는 처리 필요
```

- 장점: TTL 짧게 유지 가능
- 단점: `<video src>` 교체 시 버퍼 초기화, 재생 위치 복원 로직 필요. 구현 복잡.

---

#### ③ HLS 스트리밍 (프로덕션 방식)

FFmpeg으로 영상을 10초 단위 `.ts` 세그먼트 + `.m3u8` 매니페스트로 분할 저장.
세그먼트 단위로 짧은 토큰을 발급하여 접근 제어.

```
업로드: 영상 파일 → FFmpeg → {id}.m3u8 + {id}_000.ts, {id}_001.ts ...
재생:   클라이언트 → 매니페스트 요청 → 세그먼트 순차 요청 (각각 토큰 검증)
```

- 장점: Netflix/YouTube 방식. 화질 선택, CDN 연동, 토큰 관리 모두 자연스러움
- 단점: 파일서버에 FFmpeg 파이프라인 추가 필요, React에 HLS.js 라이브러리 필요, 인프라 복잡도 크게 증가
- 현재 범위에서는 과함. 서비스 규모가 커지면 고려.

## 4-4. 보호자원 인가 설계 (컨텍스트 기반)

### 핵심 원칙

파일 자체에 ACL(누가 볼 수 있다)을 저장하지 않는다.
파일이 **어디에 속하는지(컨텍스트)** 를 저장하고, 그 컨텍스트의 접근 규칙을 따른다.

```
파일 → CTX_TYPE='CHAT_ROOM', CTX_ID=42 → 채팅방 42번 참여자인지 확인
파일 → CTX_TYPE='LECTURE',   CTX_ID=7  → 강의 7번 구매/수강 권한 있는지 확인
파일 → CTX_TYPE='DEPT_DOC',  CTX_ID=3  → 부서 3번 소속인지 확인
파일 → CTX_TYPE='ORG_DOC',   CTX_ID=null → 원장/부서장 직급인지 확인
```

### DB 변경

```sql
ALTER TABLE CMMT_ATCH_FILE_DTL ADD CTX_TYPE VARCHAR2(20);
ALTER TABLE CMMT_ATCH_FILE_DTL ADD CTX_ID   NUMBER(19);
```

### FileCtxType enum (common 모듈)

```java
public enum FileCtxType {
    CHAT_ROOM,  // 채팅방 첨부파일 → 채팅방 참여자만
    LECTURE,    // 강의 콘텐츠 → 구매자/수강생/강사/PD
    DEPT_DOC,   // 부서 문서 → 해당 부서 소속
    ORG_DOC,    // 기관 중요 문서 → 원장/부서장 직급만
}
```

### FileAuthorizationService (common 모듈)

Admin과 REST 모듈 둘 다 이 서비스를 통해 인가 체크.

```java
public boolean canAccess(long fileId, Authentication authentication) {
    FileDto fileDto = fileMapper.findContextByFileId(fileId); // FILE_SERVER_ID로 조회
    if (fileDto == null || fileDto.getFileCtxType() == null) return false;

    String userId = authentication.getName();
    return switch (fileDto.getFileCtxType()) {
        case CHAT_ROOM -> messageMapper.isParticipant(fileDto.getFileCtxId(), userId) > 0;
        case LECTURE   -> // 미구현: 구매 여부, 수강 여부, 강사/PD 여부 체크
        case DEPT_DOC  -> // 미구현: 부서 소속 여부 체크
        default        -> true; // 임시, 실제 배포 전에 false로 변경
    };
}
```

### Mapper 쿼리 (alias 방식)

```xml
<select id="findContextByFileId" resultType="FileDto">
    select ctx_type as fileCtxType, ctx_id as fileCtxId
    from CMMT_ATCH_FILE_DTL
    where file_server_id = #{fileId}
</select>
```

> **주의**: DB에 저장되는 문자열이 enum 이름과 대소문자까지 정확히 일치해야 함.
> `'CHAT_ROOM'` → OK / `'chat_room'` → 런타임 에러

### 업로드 시 CTX_TYPE / CTX_ID 세팅

파일을 업로드하는 각 도메인 서비스에서 `FileDto` 빌드 시 컨텍스트를 넘겨야 함.

```java
// 예: 채팅 파일 업로드
FileDto fileDto = FileDto.builder()
    ...
    .fileCtxType(FileCtxType.CHAT_ROOM)
    .fileCtxId(roomSn)
    .build();
```

`FileUploadMapper.xml` INSERT에도 `CTX_TYPE`, `CTX_ID` 컬럼 추가 필요 (현재 미반영).

## 5. 구현 현황

### 파일 서버 (paste.maerchen.dev)

- Oracle 인스턴스에 Redis 설치 및 외부 접속 설정 완료
- `spring-boot-starter-data-redis` 의존성 추가
- `FileAccessTokenService` 구현 — Redis에 60초 TTL 토큰 저장/검증 (멀티유즈, 만료시 자동 차단)
- `StoredFileController` — `view`, `download` 엔드포인트에 토큰 또는 API 키 검증 추가
    - `?token=xxx` : Pre-signed URL 방식 (React 클라이언트용)
    - `X-Api-Key` 헤더 : Admin/REST 서버 직접 접근용
- Docker 실행 시 `--add-host=host.docker.internal:host-gateway` + `-e SPRING_DATA_REDIS_HOST=host.docker.internal` 로 컨테이너에서 호스트 Redis 접근

### REST 모듈

- `spring-boot-starter-data-redis` 의존성 추가
- `FileAccessTokenService` — Redis에 토큰 발급 (viewUrl, downloadUrl 반환)
- `FileController` — `POST /api/files/{fileId}/token` 엔드포인트 (JWT 인증 필요)
    - 권한 체크 후 Pre-signed URL 발급 → React 클라이언트가 파일서버 직접 접근

### Admin 모듈

- `AdminFileController` — `/admin/files/{fileId}/view`, `/admin/files/{fileId}/download`
    - API 키로 파일서버 직접 호출 후 스트림 전달 (프록시)
    - `canAccess()` — 현재 `return true` (TODO: `FileAuthorizationService` 호출로 교체)
- `ChatMessageController` — 파일 업로드 시 `msgCn`에 `/admin/files/{id}/view` 저장
- `FileDto` — `fileServerId`, `fileCtxType`(enum), `fileCtxId` 필드 추가
- `FileCtxType` — enum: `CHAT_ROOM`, `LECTURE`, `DEPT_DOC`, `ORG_DOC`
- `FileUploadService` — `fileServerId` 세팅 (`fileResponse.id()`)
- `FileUploadMapper.xml` — INSERT에 `FILE_SERVER_ID` 컬럼 추가 (CTX_TYPE/CTX_ID 미반영)
- `FileAuthorizationService` (common) — 구현 완료, `CHAT_ROOM` 케이스만 동작, 나머지 주석 처리
- `FileMapper.xml` — `findContextByFileId` alias 쿼리 추가
- `CMMT_ATCH_FILE_DTL` 테이블에 `FILE_SERVER_ID NUMBER(19)`, `CTX_TYPE VARCHAR2(20)`, `CTX_ID NUMBER(19)` 컬럼 추가
